Module 2 – API-Triggered Step Functions Orchestration

Lambda Step Functions DynamoDB SNS CloudWatch IAM

This module builds a multi-step orchestration workflow using AWS Step Functions. An external caller (API Gateway, another Lambda, or a CLI invocation) triggers a dispatcher Lambda that starts a Step Functions Standard Workflow. The state machine orchestrates two downstream steps: persisting a record to DynamoDB and sending an SNS notification. CloudWatch Logs captures execution history for both Lambda and the state machine. This pattern is the foundation for order processing, approval workflows, data transformation pipelines, and any scenario requiring durable, auditable multi-step execution.

Architecture Overview

The dispatcher Lambda is the entry point — it validates the incoming request and calls StartExecution on the state machine, passing the request payload as input. Step Functions then drives the workflow: first invoking a Lambda that writes to DynamoDB, then invoking a Lambda that publishes to SNS. Each state transition is logged to CloudWatch. If any step fails, Step Functions retries according to the configured retry policy and can route to a Catch handler for compensating actions.

  • Dispatcher Lambda: Validates input and calls sfn:StartExecution. Returns the execution ARN to the caller for status polling.
  • Step Functions State Machine: Durable orchestrator — manages state, retries, error handling, and execution history. Uses Standard Workflow for exactly-once semantics and full audit trail.
  • write_record Lambda: Receives the workflow input and writes a structured record to DynamoDB.
  • send_notify Lambda: Receives the DynamoDB write result and publishes a formatted SNS notification.
  • DynamoDB Table: Durable store for order/workflow records.
  • SNS Topic: Fan-out notification channel for downstream consumers.
  • CloudWatch Logs: Captures Lambda logs and Step Functions execution events (state transitions, errors, input/output).

DynamoDB Table

The orders table stores one item per workflow execution. The partition key is the order_id (a UUID generated by the dispatcher). A sort key of created_at (ISO-8601 timestamp) enables time-range queries per order. 

# dynamodb.tf

resource "aws_dynamodb_table" "orders" {
  name         = "order-proc-orders-${var.environment}"
  billing_mode = "PAY_PER_REQUEST"

  hash_key  = "order_id"    # partition key
  range_key = "created_at"  # sort key — enables time-range queries

  # Only declare attributes used in key schema.
  attribute {
    name = "order_id"
    type = "S"
  }

  attribute {
    name = "created_at"
    type = "S"
  }

  # TTL: automatically expire completed orders after 30 days.
  ttl {
    attribute_name = "expires_at"
    enabled        = true
  }

  point_in_time_recovery {
    enabled = true
  }
}

Worker Lambda Functions

The two worker Lambdas are invoked synchronously by Step Functions (using the Task state type with a Lambda ARN as the Resource). Each function receives the current state machine input as its event payload and returns a result that becomes the input for the next state.

write_record handler (write_record/handler.py)

"""
write_record/handler.py
Receives the workflow input, writes an order record to DynamoDB,
and returns the record for the next state.
"""
import json
import os
import boto3
from datetime import datetime, timezone

dynamodb = boto3.resource("dynamodb")
TABLE_NAME = os.environ["DYNAMODB_TABLE"]

def handler(event, context):
    order_id   = event["order_id"]
    customer   = event.get("customer", {})
    items      = event.get("items", [])
    created_at = datetime.now(timezone.utc).isoformat()

    record = {
        "order_id":   order_id,
        "created_at": created_at,
        "customer":   customer,
        "items":      items,
        "status":     "PROCESSING",
        "total":      sum(i.get("price", 0) * i.get("qty", 1) for i in items),
    }

    table = dynamodb.Table(TABLE_NAME)
    table.put_item(Item=record)
    print(f"[INFO] Wrote order {order_id} to DynamoDB")

    # Return value becomes the input to the next Step Functions state.
    return record

send_notify handler (send_notify/handler.py)

"""
send_notify/handler.py
Receives the DynamoDB record from the previous state and publishes
an SNS notification.
"""
import json
import os
import boto3

sns = boto3.client("sns")
TOPIC_ARN = os.environ["SNS_TOPIC_ARN"]

def handler(event, context):
    order_id = event["order_id"]
    total    = event.get("total", 0)
    status   = event.get("status", "UNKNOWN")

    sns.publish(
        TopicArn=TOPIC_ARN,
        Subject=f"Order {order_id} — {status}",
        Message=json.dumps({
            "order_id": order_id,
            "status":   status,
            "total":    total,
            "message":  f"Order {order_id} has been recorded successfully.",
        }, indent=2),
        MessageAttributes={
            "order_id": {"DataType": "String", "StringValue": order_id},
            "status":   {"DataType": "String", "StringValue": status},
        }
    )
    print(f"[INFO] Published SNS notification for order {order_id}")
    return {"order_id": order_id, "notification_sent": True}

Step Functions State Machine

The state machine is defined using Amazon States Language (ASL) — a JSON-based DSL. The definition argument in aws_sfn_state_machine accepts a JSON string. Using Terraform's jsonencode() function keeps the definition readable in HCL and avoids escaping issues. The workflow has two sequential Task states, each invoking a Lambda function, with retry and catch blocks for production resilience.

Amazon States Language key concepts:
  • Task state: Invokes a resource (Lambda, DynamoDB API, etc.) and waits for a result.
  • Resource ARN: For synchronous Lambda invocation, use the Lambda function ARN directly.
  • ResultPath: Controls where the task output is merged into the state input ($ = replace entire input, $.result = add as a field).
  • Retry: Automatic retry with exponential backoff on specified error types.
  • Catch: Routes to a fallback state on unrecoverable errors.
# sfn.tf

# CloudWatch log group for Step Functions execution logs.
resource "aws_cloudwatch_log_group" "sfn" {
  name              = "/aws/states/order-proc-workflow-${var.environment}"
  retention_in_days = 30
}

resource "aws_sfn_state_machine" "order_processing" {
  name     = "order-proc-workflow-${var.environment}"
  role_arn = aws_iam_role.sfn_exec.arn
  type     = "STANDARD"   # STANDARD = exactly-once, full audit trail (up to 1 year)
                           # EXPRESS  = at-least-once, high throughput, 5-min max

  # The state machine definition in Amazon States Language.
  # jsonencode() produces valid JSON from HCL maps — no manual escaping needed.
  definition = jsonencode({
    Comment = "Order processing workflow: write to DynamoDB then notify via SNS"
    StartAt = "WriteOrderRecord"

    States = {
      # ── State 1: Write to DynamoDB ──────────────────────────────────
      WriteOrderRecord = {
        Type     = "Task"
        # Synchronous Lambda invocation — SFN waits for the function to return.
        Resource = aws_lambda_function.write_record.arn
        Comment  = "Persist the order record to DynamoDB"

        # Retry on transient Lambda and service errors.
        Retry = [
          {
            ErrorEquals     = ["Lambda.ServiceException", "Lambda.AWSLambdaException",
                               "Lambda.SdkClientException", "Lambda.TooManyRequestsException"]
            IntervalSeconds = 2
            MaxAttempts     = 3
            BackoffRate     = 2.0
          }
        ]

        # On unrecoverable error, route to the failure handler.
        Catch = [
          {
            ErrorEquals = ["States.ALL"]
            Next        = "HandleFailure"
            ResultPath  = "$.error"
          }
        ]

        # Pass the entire output of this state as input to the next state.
        ResultPath = "$"
        Next       = "SendOrderNotification"
      }

      # ── State 2: Publish SNS notification ───────────────────────────
      SendOrderNotification = {
        Type     = "Task"
        Resource = aws_lambda_function.send_notify.arn
        Comment  = "Notify downstream systems via SNS"

        Retry = [
          {
            ErrorEquals     = ["Lambda.ServiceException", "Lambda.AWSLambdaException",
                               "Lambda.SdkClientException", "Lambda.TooManyRequestsException"]
            IntervalSeconds = 2
            MaxAttempts     = 3
            BackoffRate     = 2.0
          }
        ]

        Catch = [
          {
            ErrorEquals = ["States.ALL"]
            Next        = "HandleFailure"
            ResultPath  = "$.error"
          }
        ]

        ResultPath = "$"
        End        = true   # Terminal state — workflow succeeds here
      }

      # ── Failure handler ──────────────────────────────────────────────
      # In production, this could invoke a compensating Lambda or send an alert.
      HandleFailure = {
        Type  = "Fail"
        Error = "OrderProcessingFailed"
        Cause = "An unrecoverable error occurred during order processing"
      }
    }
  })

  # Enable CloudWatch logging for execution events.
  logging_configuration {
    # The log destination must include ":*" suffix for Step Functions.
    log_destination        = "${aws_cloudwatch_log_group.sfn.arn}:*"
    include_execution_data = true   # logs input/output at each state — disable in prod if sensitive
    level                  = "ERROR"  # ALL | ERROR | FATAL | OFF
  }

  depends_on = [
    aws_iam_role_policy_attachment.sfn_permissions,
    aws_cloudwatch_log_group.sfn
  ]
}

Key arguments: type (STANDARD vs EXPRESS), definition (ASL JSON string), logging_configuration block with log_destination (must end in :*), include_execution_data, and level.

STANDARD vs EXPRESS: Use STANDARD when you need exactly-once execution semantics, executions longer than 5 minutes, or a full audit trail queryable via the console and API. Use EXPRESS for high-throughput, short-duration workflows (IoT, streaming) where at-least-once is acceptable.

Dispatcher Lambda

The dispatcher Lambda is the public-facing entry point. It validates the incoming payload, generates a unique order_id, and starts a Step Functions execution. It returns the execution ARN so the caller can poll for status.

Dispatcher handler (dispatcher/handler.py)

"""
dispatcher/handler.py
Entry point for the order processing workflow.
Validates input, generates an order_id, and starts a Step Functions execution.
"""
import json
import os
import uuid
import boto3

sfn = boto3.client("stepfunctions")
STATE_MACHINE_ARN = os.environ["STATE_MACHINE_ARN"]

def handler(event, context):
    # Support both direct invocation and API Gateway proxy integration.
    if isinstance(event.get("body"), str):
        body = json.loads(event["body"])
    else:
        body = event

    # Generate a unique order ID for this execution.
    order_id = str(uuid.uuid4())
    body["order_id"] = order_id

    # Start the Step Functions execution.
    # The execution name must be unique within the state machine.
    response = sfn.start_execution(
        stateMachineArn=STATE_MACHINE_ARN,
        name=f"order-{order_id}",
        input=json.dumps(body),
    )

    print(f"[INFO] Started execution {response['executionArn']} for order {order_id}")

    return {
        "statusCode": 202,
        "body": json.dumps({
            "order_id":     order_id,
            "execution_arn": response["executionArn"],
            "status":       "ACCEPTED",
        })
    }

CloudWatch Logs for State Machine

Step Functions writes execution events (state entered, state exited, execution succeeded/failed) to the CloudWatch log group configured in logging_configuration. With include_execution_data = true, the full input and output of each state is logged — invaluable for debugging but potentially sensitive in production.

for dir in dispatcher write_record send_notify; do

Cleanup

terraform destroy
  • Step Functions executions are retained in history even after the state machine is deleted — they are not billed after deletion.
  • DynamoDB tables with data will be deleted immediately by terraform destroy unless deletion_protection_enabled = true is set.
  • CloudWatch log groups persist after Lambda/SFN deletion unless explicitly destroyed.

Common Pitfalls

Circular dependency between SFN policy and state machine: The SFN execution role policy references Lambda ARNs, and the state machine definition also references Lambda ARNs. Terraform resolves this correctly via the dependency graph — but if you try to reference the state machine ARN in the SFN policy (e.g., for self-referential executions), you will create a circular dependency. Use depends_on explicitly to break cycles.

Log destination format: The log_destination in logging_configuration must end with :* (e.g., arn:aws:logs:...:log-group:/aws/states/...:*). Omitting :* causes a silent failure where no logs are written.

  • Execution name uniqueness: Each StartExecution call requires a unique name within the state machine. Using a UUID (as shown) is the safest approach.
  • ResultPath semantics: Setting ResultPath = "$" replaces the entire state input with the task output. Use ResultPath = "$.taskResult" to merge the output into the existing input object.
  • Lambda ARN versioning in SFN: The state machine definition uses the unqualified Lambda ARN (no version suffix). Step Functions always invokes $LATEST unless you specify a version or alias ARN.
  • include_execution_data in production: Setting include_execution_data = true logs all state inputs and outputs. If your workflow processes PII or secrets, set this to false and use level = "ERROR" only.
  • STANDARD workflow limits: Standard Workflows support up to 25,000 state transitions per execution. For very long-running workflows, design states to be coarse-grained.

Best practice: Always define Retry blocks on every Task state targeting Lambda. Lambda invocations can fail transiently due to cold starts, throttling, or service errors. The recommended error set is ["Lambda.ServiceException", "Lambda.AWSLambdaException", "Lambda.SdkClientException", "Lambda.TooManyRequestsException"] with exponential backoff.
