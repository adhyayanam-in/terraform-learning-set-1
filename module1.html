Module 1 – Event-Driven Ingestion Pipeline

S3 EventBridge Lambda DynamoDB SNS CloudWatch IAM

This module builds a fully event-driven ingestion pipeline. When a file lands in an S3 bucket, EventBridge detects the creation event and invokes a Lambda function. Lambda extracts metadata from the S3 event, persists a record to DynamoDB, and publishes a notification to an SNS topic. All Lambda execution logs flow to CloudWatch Logs. This is a foundational pattern for data lake ingestion, file processing, and audit-trail systems.

Architecture Overview

The pipeline is entirely asynchronous and serverless. S3 emits an EventBridge event natively (no polling, no Lambda trigger wired directly to S3). EventBridge evaluates the event pattern and routes matching events to Lambda. Lambda is the sole compute layer — it reads the event payload, writes to DynamoDB, and fans out to SNS. 

┌─────────────────────────────────────────────────────────────────────┐
│                        AWS Account / us-east-1                      │
│                                                                     │
│  ┌──────────────┐    EventBridge     ┌──────────────────────────┐   │
│  │  S3 Bucket   │ ──────────────────▶│  EventBridge Rule        │   │
│  │  (input)     │  "Object Created"  │  source: aws.s3          │   │
│  └──────────────┘                    │  detail-type: Object      │   │
│                                      │  Created                  │   │
│                                      └────────────┬─────────────┘   │
│                                                   │ invoke          │
│                                      ┌────────────▼─────────────┐   │
│                                      │  Lambda Function          │   │
│                                      │  (process_s3_event)       │   │
│                                      └──┬──────────────┬─────────┘   │
│                                         │              │             │
│                              PutItem    │              │  Publish    │
│                         ┌───────────────▼──┐    ┌──────▼──────────┐ │
│                         │  DynamoDB Table  │    │   SNS Topic     │ │
│                         │  (s3_events)     │    │   (alerts)      │ │
│                         └──────────────────┘    └─────────────────┘ │
│                                                                     │
│  CloudWatch Logs: /aws/lambda/process-s3-event                      │
└─────────────────────────────────────────────────────────────────────┘
  • S3 Bucket: Receives uploaded files; emits native EventBridge events when EventBridge notifications are enabled.
  • EventBridge Rule: Filters for Object Created events from the specific bucket ARN and routes them to Lambda.
  • Lambda Function: Stateless compute — parses the event, writes metadata to DynamoDB, publishes to SNS.
  • DynamoDB Table: Durable, low-latency store for per-object metadata records (key, size, timestamp, ETag).
  • SNS Topic: Fan-out notification channel; subscribers (email, SQS, HTTP) receive alerts on each ingestion.
  • CloudWatch Logs: Captures all Lambda stdout/stderr for debugging and audit.
  • IAM Roles/Policies: Least-privilege execution roles for Lambda and EventBridge.

Prerequisites

Before applying this module, ensure you have:

  • Terraform CLI ≥ 1.5 installed
  • AWS credentials configured (environment variables, ~/.aws/credentials, or an IAM instance profile)
  • An AWS account with permissions to create IAM roles, Lambda functions, S3 buckets, DynamoDB tables, SNS topics, and EventBridge rules
  • A Lambda deployment package (ZIP file) — see the Lambda section below for a minimal Python example

S3 Bucket & EventBridge Notification

The S3 bucket is the entry point of the pipeline. In provider v6.x, bucket ACL configuration is managed via separate resources (aws_s3_bucket_ownership_controls and aws_s3_bucket_acl). The acl argument was removed from aws_s3_bucket in v4+. To route S3 events through EventBridge (rather than configuring Lambda triggers directly on S3), enable EventBridge notifications via aws_s3_bucket_notification with eventbridge = true. 

# s3.tf

# The primary input bucket — all uploaded objects trigger the pipeline.
resource "aws_s3_bucket" "input" {
  bucket = "etl-pipeline-input-${var.environment}-${data.aws_caller_identity.current.account_id}"

  # Prevent accidental deletion of a bucket that may contain data.
  lifecycle {
    prevent_destroy = false   # set to true in production
  }
}

# Block all public access — this bucket should never be public.
resource "aws_s3_bucket_public_access_block" "input" {
  bucket = aws_s3_bucket.input.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# Enable versioning so that overwritten objects are preserved.
resource "aws_s3_bucket_versioning" "input" {
  bucket = aws_s3_bucket.input.id

  versioning_configuration {
    status = "Enabled"
  }
}

# Enable server-side encryption with AWS-managed keys.
resource "aws_s3_bucket_server_side_encryption_configuration" "input" {
  bucket = aws_s3_bucket.input.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

# Route ALL S3 events for this bucket through EventBridge.
# This is the recommended pattern: one setting enables all event types,
# and EventBridge rules filter to exactly what you need.
resource "aws_s3_bucket_notification" "input" {
  bucket      = aws_s3_bucket.input.id
  eventbridge = true   # sends all S3 events to the default EventBridge bus
}

# Retrieve the current AWS account ID for use in bucket naming.
data "aws_caller_identity" "current" {}

Key arguments: bucket (globally unique name), eventbridge = true in aws_s3_bucket_notification (enables native EventBridge integration — no SQS or SNS intermediary needed).

Why EventBridge over direct S3 → Lambda triggers? The EventBridge pattern decouples the bucket from its consumers. You can add, remove, or change Lambda targets without modifying the bucket configuration. Multiple rules can match the same event, enabling fan-out to different processors.

IAM Roles & Policies

IAM is the most common source of failures in multi-service pipelines. Define roles and policies before the resources that depend on them. Lambda needs a trust policy allowing lambda.amazonaws.com to assume the role, plus permissions to write to DynamoDB, publish to SNS, and write CloudWatch Logs. EventBridge needs permission to invoke the Lambda function (granted via aws_lambda_permission, not an IAM policy on the rule). 

# iam.tf

# ── Lambda execution role ──────────────────────────────────────────────
# The trust policy allows the Lambda service to assume this role.
data "aws_iam_policy_document" "lambda_assume_role" {
  statement {
    effect  = "Allow"
    actions = ["sts:AssumeRole"]

    principals {
      type        = "Service"
      identifiers = ["lambda.amazonaws.com"]
    }
  }
}

resource "aws_iam_role" "lambda_exec" {
  name               = "etl-pipeline-lambda-exec-${var.environment}"
  assume_role_policy = data.aws_iam_policy_document.lambda_assume_role.json
}

# ── Lambda permissions policy ──────────────────────────────────────────
# Least-privilege: only the specific actions this Lambda needs.
data "aws_iam_policy_document" "lambda_permissions" {
  # Write structured logs to CloudWatch Logs.
  statement {
    sid    = "AllowCloudWatchLogs"
    effect = "Allow"
    actions = [
      "logs:CreateLogGroup",
      "logs:CreateLogStream",
      "logs:PutLogEvents"
    ]
    resources = ["arn:aws:logs:*:*:*"]
  }

  # Write metadata records to the DynamoDB table.
  statement {
    sid    = "AllowDynamoDBWrite"
    effect = "Allow"
    actions = [
      "dynamodb:PutItem",
      "dynamodb:UpdateItem"
    ]
    resources = [aws_dynamodb_table.s3_events.arn]
  }

  # Publish notifications to the SNS topic.
  statement {
    sid     = "AllowSNSPublish"
    effect  = "Allow"
    actions = ["sns:Publish"]
    resources = [aws_sns_topic.ingestion_alerts.arn]
  }

  # Read the triggering object from S3 (if Lambda needs to inspect content).
  statement {
    sid    = "AllowS3Read"
    effect = "Allow"
    actions = [
      "s3:GetObject",
      "s3:HeadObject"
    ]
    resources = ["${aws_s3_bucket.input.arn}/*"]
  }
}

resource "aws_iam_policy" "lambda_permissions" {
  name   = "etl-pipeline-lambda-policy-${var.environment}"
  policy = data.aws_iam_policy_document.lambda_permissions.json
}

# Attach the custom policy to the Lambda execution role.
resource "aws_iam_role_policy_attachment" "lambda_permissions" {
  role       = aws_iam_role.lambda_exec.name
  policy_arn = aws_iam_policy.lambda_permissions.arn
}

# ── EventBridge → Lambda permission ───────────────────────────────────
# aws_lambda_permission grants EventBridge the right to invoke the function.
# This is a resource-based policy on Lambda, NOT an IAM role policy.
resource "aws_lambda_permission" "allow_eventbridge" {
  statement_id  = "AllowEventBridgeInvoke"
  action        = "lambda:InvokeFunction"
  function_name = aws_lambda_function.process_s3_event.function_name
  principal     = "events.amazonaws.com"
  source_arn    = aws_cloudwatch_event_rule.s3_object_created.arn
}

Critical: Without aws_lambda_permission, EventBridge will receive a 403 when attempting to invoke Lambda, and the event will be silently dropped (or sent to a dead-letter queue if configured). Always pair every aws_cloudwatch_event_target pointing to Lambda with a corresponding aws_lambda_permission.

Lambda Function

The Lambda function is the processing core. It receives the EventBridge event (which wraps the S3 notification), extracts object metadata, writes to DynamoDB, and publishes to SNS. The deployment package is a ZIP file containing the handler code.

Minimal Python handler (save as lambda/handler.py)

"""
handler.py — S3 event processor
Receives EventBridge events for S3 Object Created, writes metadata to
DynamoDB, and publishes a notification to SNS.
"""
import json
import os
import boto3
from datetime import datetime, timezone

dynamodb = boto3.resource("dynamodb")
sns      = boto3.client("sns")

TABLE_NAME = os.environ["DYNAMODB_TABLE"]
TOPIC_ARN  = os.environ["SNS_TOPIC_ARN"]

def handler(event, context):
    # EventBridge wraps the S3 event; the S3 detail is in event["detail"]
    detail = event.get("detail", {})
    bucket = detail.get("bucket", {}).get("name", "unknown")
    key    = detail.get("object", {}).get("key", "unknown")
    size   = detail.get("object", {}).get("size", 0)
    etag   = detail.get("object", {}).get("etag", "")

    record = {
        "pk":          f"s3://{bucket}/{key}",
        "ingested_at": datetime.now(timezone.utc).isoformat(),
        "bucket":      bucket,
        "key":         key,
        "size_bytes":  size,
        "etag":        etag,
        "event_time":  event.get("time", ""),
    }

    # Persist to DynamoDB
    table = dynamodb.Table(TABLE_NAME)
    table.put_item(Item=record)
    print(f"[INFO] Wrote record for s3://{bucket}/{key}")

    # Notify via SNS
    sns.publish(
        TopicArn=TOPIC_ARN,
        Subject=f"New S3 object ingested: {key}",
        Message=json.dumps(record, indent=2),
    )
    print(f"[INFO] Published SNS notification for {key}")

    return {"statusCode": 200, "body": "OK"}

Terraform resource

# lambda.tf

resource "aws_lambda_function" "process_s3_event" {
  function_name = "etl-pipeline-process-s3-event-${var.environment}"
  description   = "Processes S3 Object Created events: writes to DynamoDB and notifies SNS"

  # The deployment package — must exist before terraform apply.
  filename         = "${path.module}/lambda_function.zip"
  source_code_hash = filebase64sha256("${path.module}/lambda_function.zip")

  # Runtime and handler reference: filename.function_name
  runtime = "python3.12"
  handler = "handler.handler"

  # Execution role — defined in iam.tf
  role = aws_iam_role.lambda_exec.arn

  # Tune these for your workload; start conservative and adjust.
  timeout      = 30    # seconds; S3 metadata ops are fast
  memory_size  = 256   # MB

  # Pass resource identifiers as environment variables — no hardcoding.
  environment {
    variables = {
      DYNAMODB_TABLE = aws_dynamodb_table.s3_events.name
      SNS_TOPIC_ARN  = aws_sns_topic.ingestion_alerts.arn
      LOG_LEVEL      = "INFO"
    }
  }

  # Ensure the log group exists before Lambda tries to write to it.
  depends_on = [
    aws_cloudwatch_log_group.lambda,
    aws_iam_role_policy_attachment.lambda_permissions
  ]
}

Key arguments: filename + source_code_hash (Terraform redeploys the function only when the ZIP changes), runtime, handler, role, environment.

Best practice: Use source_code_hash = filebase64sha256(...) so Terraform detects code changes and triggers a redeployment. Without it, changing only the ZIP contents will not cause Terraform to update the function.

DynamoDB Table

The DynamoDB table stores one item per ingested S3 object. The partition key is the full S3 URI (s3://bucket/key). Using PAY_PER_REQUEST billing mode avoids capacity planning for unpredictable ingestion rates. 

# dynamodb.tf

resource "aws_dynamodb_table" "s3_events" {
  name         = "etl-pipeline-s3-events-${var.environment}"
  billing_mode = "PAY_PER_REQUEST"   # on-demand; no capacity planning needed

  # Primary key: full S3 URI (e.g., "s3://my-bucket/path/to/file.csv")
  hash_key = "pk"

  # Only define attributes used as key schema — DynamoDB is schemaless otherwise.
  attribute {
    name = "pk"
    type = "S"   # S = String
  }

  # Automatically expire old records after 90 days.
  ttl {
    attribute_name = "ttl_epoch"
    enabled        = true
  }

  # Enable point-in-time recovery for production data protection.
  point_in_time_recovery {
    enabled = true
  }
}

Key arguments: billing_mode (PAY_PER_REQUEST or PROVISIONED), hash_key, attribute blocks (only for key attributes), ttl, point_in_time_recovery.

Common mistake: Defining attribute blocks for non-key columns causes a perpetual plan diff. Only declare attributes that are used as hash_key, range_key, or GSI/LSI keys.

SNS Topic

The SNS topic is the notification fan-out point. Subscribers (email, SQS queues, HTTP endpoints, Lambda functions) receive a message for every successfully ingested object. The topic policy restricts publishing to the Lambda execution role only. 

# sns.tf

resource "aws_sns_topic" "ingestion_alerts" {
  name         = "etl-pipeline-ingestion-alerts-${var.environment}"
  display_name = "ETL Pipeline Ingestion Alerts"

  # Encrypt messages at rest using an AWS-managed key.
  # For stricter control, use a customer-managed KMS key.
  kms_master_key_id = "alias/aws/sns"
}

# Resource-based policy: only the Lambda execution role may publish.
data "aws_iam_policy_document" "sns_topic_policy" {
  statement {
    sid    = "AllowLambdaPublish"
    effect = "Allow"
    actions = ["SNS:Publish"]

    principals {
      type        = "AWS"
      identifiers = [aws_iam_role.lambda_exec.arn]
    }

    resources = [aws_sns_topic.ingestion_alerts.arn]
  }
}

resource "aws_sns_topic_policy" "ingestion_alerts" {
  arn    = aws_sns_topic.ingestion_alerts.arn
  policy = data.aws_iam_policy_document.sns_topic_policy.json
}

# Example email subscription — replace with a real address.
resource "aws_sns_topic_subscription" "email_alert" {
  topic_arn = aws_sns_topic.ingestion_alerts.arn
  protocol  = "email"
  endpoint  = "ops-team@example.com"   # must be confirmed via email
}

EventBridge Rule & Target

The EventBridge rule filters the stream of S3 events to only Object Created events from the specific input bucket. The event_pattern uses the native EventBridge JSON pattern syntax. The target wires the rule to the Lambda function. 

# eventbridge.tf

resource "aws_cloudwatch_event_rule" "s3_object_created" {
  name        = "etl-pipeline-s3-object-created-${var.environment}"
  description = "Fires when any object is created in the ETL input bucket"

  # state replaces the deprecated is_enabled argument (provider v6+)
  state = "ENABLED"

  # Match S3 Object Created events from our specific bucket only.
  # The "resources" filter prevents this rule from firing for other buckets.
  event_pattern = jsonencode({
    source      = ["aws.s3"]
    detail-type = ["Object Created"]
    resources   = [aws_s3_bucket.input.arn]
  })
}

resource "aws_cloudwatch_event_target" "s3_to_lambda" {
  rule      = aws_cloudwatch_event_rule.s3_object_created.name
  target_id = "etl-pipeline-s3-to-lambda"
  arn       = aws_lambda_function.process_s3_event.arn

  # Optional: configure a dead-letter queue for failed invocations.
  # dead_letter_config {
  #   arn = aws_sqs_queue.dlq.arn
  # }

  # Optional: retry policy (defaults: 24h max age, 185 retries)
  retry_policy {
    maximum_event_age_in_seconds = 3600   # retry for up to 1 hour
    maximum_retry_attempts       = 3
  }
}

The reference aws_s3_bucket.input.arn creates an implicit dependency: Terraform will create the S3 bucket before the EventBridge rule. Similarly, aws_lambda_function.process_s3_event.arn ensures Lambda exists before the target.

Best practice: Always scope the resources field in the event pattern to the specific bucket ARN. Without it, the rule matches S3 events from all buckets in the account, which is almost never the intent and can cause unexpected Lambda invocations.

CloudWatch Log Group & Alarm

Lambda automatically creates a log group if one does not exist, but Terraform-managed log groups give you control over retention and encryption. Pre-creating the log group also prevents a race condition where Lambda tries to write logs before the group exists. 

# cloudwatch.tf

# Pre-create the log group so retention is enforced from the first invocation.
resource "aws_cloudwatch_log_group" "lambda" {
  name              = "/aws/lambda/${aws_lambda_function.process_s3_event.function_name}"
  retention_in_days = 30   # adjust for compliance requirements
}

# Alarm: alert when Lambda errors exceed threshold in a 5-minute window.
resource "aws_cloudwatch_metric_alarm" "lambda_errors" {
  alarm_name          = "etl-pipeline-lambda-errors-${var.environment}"
  alarm_description   = "Lambda function error rate exceeded threshold"
  comparison_operator = "GreaterThanOrEqualToThreshold"
  evaluation_periods  = 1
  metric_name         = "Errors"
  namespace           = "AWS/Lambda"
  period              = 300   # 5 minutes
  statistic           = "Sum"
  threshold           = 1     # alert on any error

  dimensions = {
    FunctionName = aws_lambda_function.process_s3_event.function_name
  }

  # Notify the ops SNS topic when the alarm fires.
  alarm_actions             = [aws_sns_topic.ingestion_alerts.arn]
  ok_actions                = [aws_sns_topic.ingestion_alerts.arn]
  insufficient_data_actions = []

  treat_missing_data = "notBreaching"   # no data = no alarm
}

# Alarm: alert on Lambda throttles (concurrency limit reached).
resource "aws_cloudwatch_metric_alarm" "lambda_throttles" {
  alarm_name          = "etl-pipeline-lambda-throttles-${var.environment}"
  alarm_description   = "Lambda function is being throttled"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 2
  metric_name         = "Throttles"
  namespace           = "AWS/Lambda"
  period              = 60
  statistic           = "Sum"
  threshold           = 0

  dimensions = {
    FunctionName = aws_lambda_function.process_s3_event.function_name
  }

  alarm_actions      = [aws_sns_topic.ingestion_alerts.arn]
  treat_missing_data = "notBreaching"
}

Deploying & Testing

Deploy

# 1. Package the Lambda handler
cd lambda && zip ../lambda_function.zip handler.py && cd ..

# 2. Initialize the Terraform working directory
terraform init

# 3. Preview the execution plan — review every resource before applying
terraform plan -out=tfplan

# 4. Apply the plan
terraform apply tfplan

Trigger the pipeline

# Upload a test file to the S3 bucket (replace with your bucket name from outputs)
BUCKET=$(terraform output -raw input_bucket_name)
echo '{"test": true}' | aws s3 cp - s3://${BUCKET}/test/sample.json

Verify results

# Check DynamoDB for the ingested record
TABLE=$(terraform output -raw dynamodb_table_name)
aws dynamodb scan --table-name ${TABLE} --query "Items[0]"

# Tail Lambda logs in CloudWatch
FUNCTION=$(terraform output -raw lambda_function_name)
aws logs tail /aws/lambda/${FUNCTION} --follow

# Check EventBridge rule metrics (invocations, failed invocations)
aws cloudwatch get-metric-statistics \
  --namespace AWS/Events \
  --metric-name TriggeredRules \
  --dimensions Name=RuleName,Value=etl-pipeline-s3-object-created-dev \
  --start-time $(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --period 300 \
  --statistics Sum

Cleanup

# Remove all resources created by this module
terraform destroy

S3 bucket deletion: terraform destroy will fail if the S3 bucket contains objects. Empty the bucket first:

BUCKET=$(terraform output -raw input_bucket_name)

# Delete all object versions (required if versioning is enabled)
aws s3api delete-objects \
  --bucket ${BUCKET} \
  --delete "$(aws s3api list-object-versions \
    --bucket ${BUCKET} \
    --query '{Objects: Versions[].{Key:Key,VersionId:VersionId}}')"

# Then destroy
terraform destroy

Common Pitfalls

Missing aws_lambda_permission: EventBridge cannot invoke Lambda without a resource-based policy granting lambda:InvokeFunction to events.amazonaws.com. The IAM role policy on Lambda is not sufficient — you need both.

S3 EventBridge not enabled: If aws_s3_bucket_notification with eventbridge = true is missing, S3 will not emit events to EventBridge and the pipeline will never trigger. Verify with: aws s3api get-bucket-notification-configuration --bucket <name>

  • Event pattern mismatch: The detail-type must be exactly "Object Created" (with a capital C). Typos silently drop all events.
  • Lambda timeout: The default Lambda timeout is 3 seconds. If your handler does DynamoDB writes and SNS publishes, set timeout to at least 15–30 seconds.
  • DynamoDB attribute over-definition: Only define attribute blocks for key attributes. Defining non-key attributes causes a perpetual plan diff.
  • SNS subscription confirmation: Email subscriptions require manual confirmation. The subscription will remain in PendingConfirmation state until the recipient clicks the confirmation link.
  • Log group race condition: Without depends_on = [aws_cloudwatch_log_group.lambda] in the Lambda resource, Lambda may create the log group without the retention policy you defined.
  • Region consistency: All resources must be in the same region. Cross-region EventBridge routing requires additional configuration not covered here.

Best practice: Use terraform plan output to verify the dependency graph before every apply. Look for any known after apply values in critical fields like ARNs — these indicate a resource hasn't been created yet and Terraform will resolve them in the correct order.
