Module 3 – Scheduled Data Processing with EventBridge and Lambda

EventBridge Lambda S3 DynamoDB SNS CloudWatch IAM

This module implements a scheduled batch processing pipeline. An EventBridge scheduled rule fires on a cron expression, invoking a Lambda function that reads data files from S3, processes them (aggregates, transforms, or validates), writes summary results to DynamoDB, and optionally publishes a completion or failure notification to SNS. A CloudWatch alarm monitors for Lambda errors and throttles, routing alerts to the same SNS topic. This pattern is common for nightly ETL jobs, periodic report generation, data quality checks, and cost-optimization sweeps.

Architecture Overview

The schedule is the sole trigger — no human or application action is required. EventBridge evaluates the cron expression in UTC and delivers a synthetic event to Lambda. Lambda lists objects in an S3 prefix, processes each file, aggregates results, and writes a summary record to DynamoDB. On success, it publishes a completion notification to SNS. On any unhandled exception, the CloudWatch alarm fires and routes an alert to the same SNS topic.

  • EventBridge Scheduled Rule: Cron-based trigger; fires at a fixed UTC schedule. No event source mapping or polling required.
  • Lambda Function: Stateless batch processor — lists S3 objects, reads and aggregates data, writes results, publishes notifications.
  • S3 Bucket: Holds the raw data files to be processed. Lambda reads from a date-partitioned prefix (e.g., data/YYYY/MM/DD/).
  • DynamoDB Table: Stores one summary record per job run (run ID, timestamp, files processed, record count, status).
  • SNS Topic: Receives both success notifications (from Lambda) and failure alerts (from CloudWatch alarms).
  • CloudWatch Logs & Alarms: Captures Lambda output; alarms on errors and throttles.

Prerequisites & Cron Syntax

EventBridge uses a six-field cron syntax that differs slightly from standard Unix cron. All times are in UTC. The ? wildcard is required in either the day-of-month or day-of-week field (not both).

ExpressionMeaning
cron(0 2 * * ? *)Every day at 02:00 UTC
cron(0 6 ? * MON-FRI *)Weekdays at 06:00 UTC
cron(0 0 1 * ? *)First day of every month at midnight UTC
cron(0/15 * * * ? *)Every 15 minutes
rate(1 hour)Every hour (simpler rate syntax)
rate(5 minutes)Every 5 minutes

EventBridge cron vs Unix cron: EventBridge cron has six fields: Minutes Hours Day-of-month Month Day-of-week Year. The ? character means "no specific value" and is required in either day-of-month or day-of-week. Using standard 5-field Unix cron syntax will cause a validation error.

S3 Data Bucket

The data bucket holds raw input files organized by date partition (data/YYYY/MM/DD/filename.csv). Lambda lists objects under the current date's prefix and processes each one. Lifecycle rules automatically transition old data to cheaper storage tiers.

  bucket = "batch-proc-data-lake-${var.environment}-${data.aws_caller_identity.current.account_id}"
}

  bucket                  = aws_s3_bucket.data_lake.id
  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

resource "aws_s3_bucket_versioning" "data_lake" {
  bucket = aws_s3_bucket.data_lake.id
  versioning_configuration {
    status = "Enabled"
  }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "data_lake" {
  bucket = aws_s3_bucket.data_lake.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

# Lifecycle: move data older than 30 days to Intelligent-Tiering,
# expire after 365 days. Adjust for your retention requirements.
resource "aws_s3_bucket_lifecycle_configuration" "data_lake" {
  bucket = aws_s3_bucket.data_lake.id

  rule {
    id     = "data-tiering"
    status = "Enabled"

    filter {
      prefix = var.data_prefix
    }

    transition {
      days          = 30
      storage_class = "INTELLIGENT_TIERING"
    }

    expiration {
      days = 365
    }

    # Also expire non-current versions to control storage costs.
    noncurrent_version_expiration {
      noncurrent_days = 30
    }
  }
}

Lambda Function

The batch processor Lambda is designed for reliability: it catches all exceptions, writes a failure record to DynamoDB, and publishes a failure notification to SNS before re-raising. This ensures the job result is always recorded, even on failure, and the CloudWatch alarm provides a second layer of alerting.

Handler (batch_processor/handler.py)

"""
batch_processor/handler.py
Scheduled batch processor:
  1. Lists S3 objects under today's date prefix.
  2. Reads and aggregates each CSV file (counts rows, sums a value column).
  3. Writes a summary record to DynamoDB.
  4. Publishes a completion or failure notification to SNS.
"""
import csv
import io
import json
import os
import uuid
import boto3
from datetime import datetime, timezone

s3       = boto3.client("s3")
dynamodb = boto3.resource("dynamodb")
sns      = boto3.client("sns")

BUCKET_NAME    = os.environ["BUCKET_NAME"]
DATA_PREFIX    = os.environ["DATA_PREFIX"]
DYNAMODB_TABLE = os.environ["DYNAMODB_TABLE"]
SNS_TOPIC_ARN  = os.environ["SNS_TOPIC_ARN"]

def handler(event, context):
    run_id    = str(uuid.uuid4())
    now       = datetime.now(timezone.utc)
    date_str  = now.strftime("%Y/%m/%d")
    prefix    = f"{DATA_PREFIX}{date_str}/"

    print(f"[INFO] Run {run_id} — processing prefix s3://{BUCKET_NAME}/{prefix}")

    total_rows   = 0
    total_value  = 0.0
    files_processed = 0
    status       = "SUCCESS"
    error_detail = None

    try:
        # List all objects under today's date prefix.
        paginator = s3.get_paginator("list_objects_v2")
        pages = paginator.paginate(Bucket=BUCKET_NAME, Prefix=prefix)

        for page in pages:
            for obj in page.get("Contents", []):
                key = obj["Key"]
                if not key.endswith(".csv"):
                    continue

                # Read and parse the CSV file.
                response = s3.get_object(Bucket=BUCKET_NAME, Key=key)
                content  = response["Body"].read().decode("utf-8")
                reader   = csv.DictReader(io.StringIO(content))

                for row in reader:
                    total_rows  += 1
                    # Assumes a "value" column; adapt to your schema.
                    total_value += float(row.get("value", 0))

                files_processed += 1
                print(f"[INFO] Processed {key}")

    except Exception as exc:
        status       = "FAILED"
        error_detail = str(exc)
        print(f"[ERROR] Batch run {run_id} failed: {exc}")

    # Always write the job result to DynamoDB — even on failure.
    table = dynamodb.Table(DYNAMODB_TABLE)
    record = {
        "run_id":          run_id,
        "run_date":        now.strftime("%Y-%m-%d"),
        "executed_at":     now.isoformat(),
        "status":          status,
        "files_processed": files_processed,
        "total_rows":      total_rows,
        "total_value":     str(round(total_value, 2)),  # DynamoDB requires Decimal or str for floats
        "s3_prefix":       prefix,
    }
    if error_detail:
        record["error"] = error_detail

    table.put_item(Item=record)
    print(f"[INFO] Wrote job result for run {run_id}: {status}")

    # Publish SNS notification.
    subject = f"Batch Job {status}: {run_id[:8]}"
    sns.publish(
        TopicArn=SNS_TOPIC_ARN,
        Subject=subject,
        Message=json.dumps(record, indent=2),
    )

    if status == "FAILED":
        # Re-raise so Lambda marks the invocation as an error,
        # which triggers the CloudWatch alarm.
        raise RuntimeError(f"Batch run {run_id} failed: {error_detail}")

    return {"run_id": run_id, "status": status, "files_processed": files_processed}

Terraform resource

# lambda.tf

resource "aws_lambda_function" "batch_processor" {
  function_name    = "batch-proc-processor-${var.environment}"
  description      = "Scheduled batch processor: reads S3, aggregates, writes DynamoDB, notifies SNS"
  filename         = "${path.module}/batch_processor.zip"
  source_code_hash = filebase64sha256("${path.module}/batch_processor.zip")
  runtime          = "python3.12"
  handler          = "handler.handler"
  role             = aws_iam_role.batch_lambda.arn

  # Batch jobs processing many files may run for several minutes.
  # Lambda maximum timeout is 900 seconds (15 minutes).
  timeout     = 300   # 5 minutes; adjust based on expected data volume
  memory_size = 512   # More memory = more CPU; useful for CSV parsing

  environment {
    variables = {
      BUCKET_NAME    = aws_s3_bucket.data_lake.bucket
      DATA_PREFIX    = var.data_prefix
      DYNAMODB_TABLE = aws_dynamodb_table.job_results.name
      SNS_TOPIC_ARN  = aws_sns_topic.batch_alerts.arn
      LOG_LEVEL      = "INFO"
    }
  }

  depends_on = [
    aws_cloudwatch_log_group.batch_lambda,
    aws_iam_role_policy_attachment.batch_lambda_permissions
  ]
}

Memory and timeout sizing: Lambda allocates CPU proportional to memory. For I/O-bound workloads (S3 reads, DynamoDB writes), 512 MB is a good starting point. For CPU-bound processing (parsing, compression, encryption), consider 1024–3008 MB. Always set timeout well above your p99 execution time — a timeout is an error from Lambda's perspective and will trigger your CloudWatch alarm.

DynamoDB Results Table

The job results table stores one record per batch run. The partition key is run_date (YYYY-MM-DD) and the sort key is run_id (UUID). This schema allows querying all runs for a given date with a Query operation, which is far more efficient than a Scan. 

# dynamodb.tf

resource "aws_dynamodb_table" "job_results" {
  name         = "batch-proc-job-results-${var.environment}"
  billing_mode = "PAY_PER_REQUEST"

  # Partition by date — enables efficient queries like "all runs on 2026-07-09"
  hash_key  = "run_date"
  range_key = "run_id"

  attribute {
    name = "run_date"
    type = "S"
  }

  attribute {
    name = "run_id"
    type = "S"
  }

  # Expire old job records after 90 days.
  ttl {
    attribute_name = "ttl_epoch"
    enabled        = true
  }

  point_in_time_recovery {
    enabled = true
  }
}

A single SNS topic receives both Lambda-published completion/failure messages and CloudWatch alarm notifications. This consolidates all batch job alerting into one channel.

EventBridge Scheduled Rule

The EventBridge rule uses a schedule_expression instead of an event_pattern. Scheduled rules can only be placed on the default event bus. The rule delivers a synthetic event to Lambda — the event payload contains metadata about the rule invocation but no application-specific data (the Lambda reads its configuration from environment variables). 

# eventbridge.tf

resource "aws_cloudwatch_event_rule" "daily_batch" {
  name                = "batch-proc-daily-schedule-${var.environment}"
  description         = "Triggers the batch processor Lambda on a schedule"
  schedule_expression = var.schedule_expression   # e.g., "cron(0 2 * * ? *)"

  # state replaces the deprecated is_enabled argument.
  state = "ENABLED"

  # Note: schedule_expression and event_pattern are mutually exclusive.
  # Scheduled rules are always placed on the default event bus.
}

resource "aws_cloudwatch_event_target" "schedule_to_lambda" {
  rule      = aws_cloudwatch_event_rule.daily_batch.name
  target_id = "batch-proc-daily-lambda-target"
  arn       = aws_lambda_function.batch_processor.arn

  # Optional: pass a static JSON input to override the default EventBridge event.
  # Useful for parameterizing the Lambda without changing environment variables.
  # input = jsonencode({ "override_prefix": "data/backfill/" })

  # Retry policy: for scheduled jobs, limit retries to avoid duplicate processing.
  retry_policy {
    maximum_event_age_in_seconds = 1800   # retry for up to 30 minutes
    maximum_retry_attempts       = 2
  }
}

Idempotency for scheduled jobs: EventBridge may invoke Lambda more than once for a single scheduled event (at-least-once delivery). Design your Lambda handler to be idempotent — use the run_id UUID as a DynamoDB condition expression to prevent duplicate writes, or use a DynamoDB conditional put: ConditionExpression="attribute_not_exists(run_id)".

CloudWatch Log Group & Alarms

For scheduled batch jobs, observability is critical — there is no human in the loop to notice a silent failure. Define alarms for Lambda errors, throttles, and duration (to catch runaway jobs). All alarms route to the SNS topic. 

# cloudwatch.tf

# Pre-create the log group with a retention policy.
resource "aws_cloudwatch_log_group" "batch_lambda" {
  name              = "/aws/lambda/${aws_lambda_function.batch_processor.function_name}"
  retention_in_days = 90   # keep batch job logs for 90 days for audit
}

# ── Alarm: Lambda errors ───────────────────────────────────────────────
# Fires if the Lambda function returns any error (exception or timeout).
resource "aws_cloudwatch_metric_alarm" "batch_lambda_errors" {
  alarm_name          = "batch-proc-lambda-errors-${var.environment}"
  alarm_description   = "Batch processor Lambda returned an error"
  comparison_operator = "GreaterThanOrEqualToThreshold"
  evaluation_periods  = 1
  metric_name         = "Errors"
  namespace           = "AWS/Lambda"
  period              = 300   # 5-minute window matches the schedule granularity
  statistic           = "Sum"
  threshold           = 1

  dimensions = {
    FunctionName = aws_lambda_function.batch_processor.function_name
  }

  alarm_actions             = [aws_sns_topic.batch_alerts.arn]
  ok_actions                = [aws_sns_topic.batch_alerts.arn]
  insufficient_data_actions = []
  treat_missing_data        = "notBreaching"
}

# ── Alarm: Lambda throttles ────────────────────────────────────────────
# Fires if Lambda is throttled (concurrent execution limit reached).
resource "aws_cloudwatch_metric_alarm" "batch_lambda_throttles" {
  alarm_name          = "batch-proc-lambda-throttles-${var.environment}"
  alarm_description   = "Batch processor Lambda is being throttled"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 1
  metric_name         = "Throttles"
  namespace           = "AWS/Lambda"
  period              = 300
  statistic           = "Sum"
  threshold           = 0

  dimensions = {
    FunctionName = aws_lambda_function.batch_processor.function_name
  }

  alarm_actions      = [aws_sns_topic.batch_alerts.arn]
  treat_missing_data = "notBreaching"
}

# ── Alarm: Lambda duration approaching timeout ─────────────────────────
# Fires if average duration exceeds 80% of the configured timeout (300s).
# This gives you time to investigate before the job starts timing out.
resource "aws_cloudwatch_metric_alarm" "batch_lambda_duration" {
  alarm_name          = "batch-proc-lambda-duration-${var.environment}"
  alarm_description   = "Batch processor Lambda duration is approaching timeout"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 1
  metric_name         = "Duration"
  namespace           = "AWS/Lambda"
  period              = 300
  statistic           = "Maximum"
  threshold           = 240000   # 240,000 ms = 240 seconds (80% of 300s timeout)
  unit                = "Milliseconds"

  dimensions = {
    FunctionName = aws_lambda_function.batch_processor.function_name
  }

  alarm_actions      = [aws_sns_topic.batch_alerts.arn]
  treat_missing_data = "notBreaching"
}

# ── Alarm: No invocations (missed schedule) ────────────────────────────
# Fires if the Lambda was NOT invoked in the expected window.
# This catches cases where the EventBridge rule was accidentally disabled.
# Adjust the period to match your schedule (86400s = 24 hours for daily jobs).
resource "aws_cloudwatch_metric_alarm" "batch_lambda_no_invocations" {
  alarm_name          = "batch-proc-lambda-no-invocations-${var.environment}"
  alarm_description   = "Batch processor Lambda was not invoked — schedule may be broken"
  comparison_operator = "LessThanThreshold"
  evaluation_periods  = 1
  metric_name         = "Invocations"
  namespace           = "AWS/Lambda"
  period              = 86400   # 24-hour window for a daily job
  statistic           = "Sum"
  threshold           = 1

  dimensions = {
    FunctionName = aws_lambda_function.batch_processor.function_name
  }

  alarm_actions      = [aws_sns_topic.batch_alerts.arn]
  # treat_missing_data = "breaching" ensures the alarm fires if there is no data
  # (i.e., Lambda was never invoked in the period).
  treat_missing_data = "breaching"
}

The "missed schedule" alarm (batch_lambda_no_invocations) is one of the most valuable alarms for scheduled jobs. Without it, a disabled EventBridge rule or a misconfigured schedule will silently skip job runs indefinitely. Set treat_missing_data = "breaching" so the alarm fires when there is no invocation data at all.

Common Pitfalls

Six-field cron syntax: EventBridge requires a six-field cron expression (Minutes Hours Day-of-month Month Day-of-week Year). Standard Unix cron has five fields. Using five fields will cause a Terraform apply error: ValidationException: Parameter ScheduleExpression is not valid.

Scheduled rules on non-default event buses: The schedule_expression argument only works on the default event bus. If you specify a custom event_bus_name, you must use event_pattern instead. Terraform will not validate this constraint — it will apply successfully but the rule will never fire.

  • Lambda timeout for large datasets: Lambda has a maximum timeout of 900 seconds. For very large datasets, consider breaking the job into smaller chunks using Step Functions Map state or SQS-based fan-out.
  • Duplicate processing: EventBridge guarantees at-least-once delivery. Use DynamoDB conditional writes (attribute_not_exists(run_id)) to prevent duplicate records if Lambda is invoked twice for the same scheduled event.
  • S3 ListObjects pagination: list_objects_v2 returns at most 1,000 objects per page. Always use a paginator (as shown in the handler) to process all objects in a prefix.
  • Float precision in DynamoDB: DynamoDB's N type uses arbitrary-precision decimals. Python's float can introduce precision errors. Use Python's Decimal type or store floats as strings.
  • Cold start latency: For scheduled jobs, cold start latency is usually acceptable. If your job has strict SLA requirements, consider using Provisioned Concurrency on the Lambda function.
  • CloudWatch alarm period alignment: The "no invocations" alarm uses a 24-hour period. CloudWatch evaluates alarms at the end of each period. For a daily job, the alarm may not fire until up to 24 hours after the missed run. For tighter SLAs, use a shorter period and adjust the threshold accordingly.

Best practice — environment-specific schedules: Use a Terraform variable for schedule_expression and set it to rate(5 minutes) in dev/test environments for rapid iteration, and cron(0 2 * * ? *) in production. This avoids waiting 24 hours to test a daily job.
