Observability Architecture

The observability layer sits alongside the application resources — it does not modify them. CloudWatch collects metrics from Lambda, Step Functions, and DynamoDB automatically. Log metric filters extract custom signals from Lambda log output. Alarms evaluate metric data against thresholds and transition between OK, ALARM, and INSUFFICIENT_DATA states. Composite alarms combine multiple child alarms to reduce noise. All alarm state changes route to SNS topics, which fan out to email, PagerDuty, Slack, or any HTTP endpoint.

Prerequisites

This module assumes the application resources from Modules 1–3 already exist (or equivalent Lambda functions, Step Functions state machines, and DynamoDB tables). The observability resources reference those application resources by name/ARN via Terraform data sources or variables. You can apply this module independently by providing the resource names as variables. 

# versions.tf
terraform {
  required_version = ">= 1.5"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 6.0"
    }
  }
}

provider "aws" {
  region = var.aws_region

  default_tags {
    tags = {
      Project     = "observability"
      Environment = var.environment
      ManagedBy   = "terraform"
    }
  }
}

variable "aws_region"  { default = "us-east-1" }
variable "environment" { default = "dev" }

# Names of application resources to monitor — passed in from the application stack
# or looked up via data sources.
variable "lambda_function_names" {
  description = "List of Lambda function names to monitor"
  type        = list(string)
  default     = [
    "etl-pipeline-process-s3-event-dev",
    "batch-proc-processor-dev",
    "order-proc-dispatcher-dev",
    "order-proc-write-record-dev",
    "order-proc-send-notify-dev"
  ]
}

variable "sfn_state_machine_name" {
  description = "Step Functions state machine name to monitor"
  default     = "order-proc-workflow-dev"
}

variable "dynamodb_table_names" {
  description = "List of DynamoDB table names to monitor"
  type        = list(string)
  default     = [
    "etl-pipeline-s3-events-dev",
    "order-proc-orders-dev",
    "batch-proc-job-results-dev"
  ]
}

variable "ops_email" {
  description = "Email address for operational alerts"
  default     = "ops-team@example.com"

  name = var.sfn_state_machine_name
}

SNS Notification Channels

Separate SNS topics for critical and warning alerts allow different routing rules. Critical alerts (Lambda errors, SFN failures) go to PagerDuty or on-call rotation. Warning alerts (throttles, high latency) go to a Slack channel or email digest. Both topics allow CloudWatch to publish alarm state changes. 

# sns.tf

# ── Critical alerts topic ──────────────────────────────────────────────
# For alarms that require immediate human response.
resource "aws_sns_topic" "ops_critical" {
  name              = "observability-ops-critical-${var.environment}"
  display_name      = "OPS CRITICAL Alerts"
  kms_master_key_id = "alias/aws/sns"
}

# ── Warning alerts topic ───────────────────────────────────────────────
# For alarms that need attention but are not immediately service-impacting.
resource "aws_sns_topic" "ops_warning" {
  name              = "observability-ops-warning-${var.environment}"
  display_name      = "OPS Warning Alerts"
  kms_master_key_id = "alias/aws/sns"
}

# Allow CloudWatch to publish to both topics.
data "aws_iam_policy_document" "allow_cloudwatch_publish" {
  statement {
    sid     = "AllowCloudWatchAlarms"
    effect  = "Allow"
    actions = ["SNS:Publish"]
    principals {
      type        = "Service"
      identifiers = ["cloudwatch.amazonaws.com"]
    }
    # Restrict to alarms in this account only.
    resources = [
      aws_sns_topic.ops_critical.arn,
      aws_sns_topic.ops_warning.arn
    ]
    condition {
      test     = "StringEquals"
      variable = "aws:SourceAccount"
      values   = [data.aws_caller_identity.current.account_id]
    }
  }
}

resource "aws_sns_topic_policy" "ops_critical" {
  arn    = aws_sns_topic.ops_critical.arn
  policy = data.aws_iam_policy_document.allow_cloudwatch_publish.json
}

resource "aws_sns_topic_policy" "ops_warning" {
  arn    = aws_sns_topic.ops_warning.arn
  policy = data.aws_iam_policy_document.allow_cloudwatch_publish.json
}

# Email subscriptions — must be confirmed by the recipient.
resource "aws_sns_topic_subscription" "critical_email" {
  topic_arn = aws_sns_topic.ops_critical.arn
  protocol  = "email"
  endpoint  = var.ops_email
}

resource "aws_sns_topic_subscription" "warning_email" {
  topic_arn = aws_sns_topic.ops_warning.arn
  protocol  = "email"
  endpoint  = var.ops_email
}

data "aws_caller_identity" "current" {}

CloudWatch Log Groups

Terraform-managed log groups enforce retention policies and encryption. Without explicit management, Lambda creates log groups with infinite retention — a compliance and cost risk. Use for_each to manage log groups for all Lambda functions from a single resource block. 

# log_groups.tf

# Manage log groups for all monitored Lambda functions.
# for_each creates one aws_cloudwatch_log_group per function name.
resource "aws_cloudwatch_log_group" "lambda_functions" {
  for_each = toset(var.lambda_function_names)

  name              = "/aws/lambda/${each.value}"
  retention_in_days = 30

  # Prevent Terraform from failing if Lambda already created the group.
  lifecycle {
    ignore_changes = [tags]
  }
}

# Log group for the Step Functions state machine.
resource "aws_cloudwatch_log_group" "sfn_state_machine" {
  name              = "/aws/states/${var.sfn_state_machine_name}"
  retention_in_days = 30
}

# ── Log Insights query examples ────────────────────────────────────────
# These are not Terraform resources, but save them as saved queries
# using aws_cloudwatch_query_definition.

resource "aws_cloudwatch_query_definition" "lambda_errors" {
  name = "observability/lambda-errors-last-hour"

  # Apply to all Lambda log groups.
  log_group_names = [
    for name in var.lambda_function_names : "/aws/lambda/${name}"
  ]

  query_string = <<-EOT
    fields @timestamp, @message, @logStream
    | filter @message like /\[ERROR\]/
    | sort @timestamp desc
    | limit 50
  EOT
}

resource "aws_cloudwatch_query_definition" "sfn_failed_executions" {
  name = "observability/sfn-failed-executions"

  log_group_names = [aws_cloudwatch_log_group.sfn_state_machine.name]

  query_string = <<-EOT
    fields @timestamp, type, details.error, details.cause
    | filter type = "ExecutionFailed"
    | sort @timestamp desc
    | limit 20
  EOT
}

Log Metric Filters

Log metric filters scan incoming log events for a pattern and increment a custom CloudWatch metric when a match is found. This lets you alarm on application-level signals (e.g., [ERROR] log lines) that are not captured by AWS service metrics. The metric is created automatically when the filter is applied — no separate metric resource is needed. 

# metric_filters.tf

# ── Application error metric filter ───────────────────────────────────
# Counts log lines containing "[ERROR]" across all Lambda functions.
# Creates a custom metric in the "ETLPipeline" namespace.
resource "aws_cloudwatch_log_metric_filter" "app_errors" {
  for_each = toset(var.lambda_function_names)

  name           = "app-errors-${each.value}"
  log_group_name = "/aws/lambda/${each.value}"

  # Pattern: match any log line containing the literal string [ERROR].
  # CloudWatch filter patterns are NOT regex — use literal strings or
  # JSON field matching for structured logs.
  pattern = "[ERROR]"

  metric_transformation {
    name          = "ApplicationErrors"
    namespace     = "ETLPipeline/Lambda"
    value         = "1"          # increment by 1 for each matching log line
    default_value = "0"          # emit 0 when no matches (enables "no data" detection)
    unit          = "Count"
  }

  depends_on = [aws_cloudwatch_log_group.lambda_functions]
}

# ── Application warning metric filter ─────────────────────────────────
resource "aws_cloudwatch_log_metric_filter" "app_warnings" {
  for_each = toset(var.lambda_function_names)

  name           = "app-warnings-${each.value}"
  log_group_name = "/aws/lambda/${each.value}"
  pattern        = "[WARN]"

  metric_transformation {
    name          = "ApplicationWarnings"
    namespace     = "ETLPipeline/Lambda"
    value         = "1"
    default_value = "0"
    unit          = "Count"
  }

  depends_on = [aws_cloudwatch_log_group.lambda_functions]
}

# ── Structured JSON log filter ─────────────────────────────────────────
# If your Lambda emits structured JSON logs (e.g., via aws-lambda-powertools),
# you can filter on specific JSON fields.
# Example: match logs where "level" = "ERROR" in JSON output.
resource "aws_cloudwatch_log_metric_filter" "structured_errors" {
  for_each = toset(var.lambda_function_names)

  name           = "structured-errors-${each.value}"
  log_group_name = "/aws/lambda/${each.value}"

  # JSON filter pattern: matches log events where the "level" field equals "ERROR".
  # The { } syntax enables JSON field matching in CloudWatch filter patterns.
  pattern = "{ $.level = \"ERROR\" }"

  metric_transformation {
    name          = "StructuredErrors"
    namespace     = "ETLPipeline/Lambda"
    value         = "1"
    default_value = "0"
    unit          = "Count"
  }

  depends_on = [aws_cloudwatch_log_group.lambda_functions]
}

CloudWatch filter pattern syntax: CloudWatch uses its own pattern language — not regex. Key patterns: [ERROR] matches the literal string; { $.field = "value" } matches JSON fields; ?ERROR ?WARN matches lines containing either word. See the CloudWatch filter pattern syntax docs for the full reference.

Lambda Alarms

The following alarms cover the most important Lambda operational signals. Each alarm is defined per function using for_each to avoid repetition.

AWS/Lambda
Errors
Count of invocations that resulted in a function error (exception, timeout, or OOM). Any non-zero value in a 5-minute window is critical.
AWS/Lambda
Throttles
Count of invocations rejected due to concurrency limits. Indicates the function needs reserved concurrency or the account limit needs raising.
AWS/Lambda
Duration
Execution time in milliseconds. Alarm on p99 or Maximum approaching the configured timeout to catch runaway executions before they fail.
AWS/Lambda
ConcurrentExecutions
Number of function instances running simultaneously. Alarm when approaching the account or function-level concurrency limit.
# lambda_alarms.tf

# ── Lambda Errors alarm (per function) ────────────────────────────────
resource "aws_cloudwatch_metric_alarm" "lambda_errors" {
  for_each = toset(var.lambda_function_names)

  alarm_name          = "lambda-errors-${each.value}"
  alarm_description   = "Lambda function ${each.value} returned one or more errors"
  comparison_operator = "GreaterThanOrEqualToThreshold"
  evaluation_periods  = 1
  metric_name         = "Errors"
  namespace           = "AWS/Lambda"
  period              = 300   # 5-minute evaluation window
  statistic           = "Sum"
  threshold           = 1     # alert on any single error

  dimensions = {
    FunctionName = each.value
  }

  # Route to critical topic — Lambda errors are service-impacting.
  alarm_actions             = [aws_sns_topic.ops_critical.arn]
  ok_actions                = [aws_sns_topic.ops_critical.arn]
  insufficient_data_actions = []
  treat_missing_data        = "notBreaching"
}

# ── Lambda Throttles alarm (per function) ─────────────────────────────
resource "aws_cloudwatch_metric_alarm" "lambda_throttles" {
  for_each = toset(var.lambda_function_names)

  alarm_name          = "lambda-throttles-${each.value}"
  alarm_description   = "Lambda function ${each.value} is being throttled"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 2    # require 2 consecutive periods to reduce noise
  metric_name         = "Throttles"
  namespace           = "AWS/Lambda"
  period              = 60
  statistic           = "Sum"
  threshold           = 0

  dimensions = {
    FunctionName = each.value
  }

  # Throttles are a warning — they degrade performance but don't lose data
  # (EventBridge and SFN retry throttled invocations).
  alarm_actions      = [aws_sns_topic.ops_warning.arn]
  ok_actions         = [aws_sns_topic.ops_warning.arn]
  treat_missing_data = "notBreaching"
}

# ── Lambda Duration alarm (per function) ──────────────────────────────
# Alarms when the maximum execution duration in a period exceeds 80% of timeout.
# Requires knowing the timeout per function — use a map variable for production.
resource "aws_cloudwatch_metric_alarm" "lambda_duration" {
  for_each = toset(var.lambda_function_names)

  alarm_name          = "lambda-duration-${each.value}"
  alarm_description   = "Lambda function ${each.value} duration is approaching timeout"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 1
  metric_name         = "Duration"
  namespace           = "AWS/Lambda"
  period              = 300
  # Use Maximum to catch the worst-case invocation in the period.
  # Use p99 (extended statistics) for a more representative view.
  extended_statistic  = "p99"
  # Threshold: 24,000 ms = 24 seconds (80% of a 30-second timeout).
  # Adjust per function based on its configured timeout.
  threshold           = 24000
  unit                = "Milliseconds"

  dimensions = {
    FunctionName = each.value
  }

  alarm_actions      = [aws_sns_topic.ops_warning.arn]
  treat_missing_data = "notBreaching"
}

# ── Custom application error alarm (from log metric filter) ───────────
# Alarms on the custom "ApplicationErrors" metric extracted from logs.
# This catches application-level errors that don't cause Lambda to fail
# (e.g., a caught exception that logs [ERROR] but returns HTTP 200).
resource "aws_cloudwatch_metric_alarm" "app_errors" {
  for_each = toset(var.lambda_function_names)

  alarm_name          = "app-errors-${each.value}"
  alarm_description   = "Application-level [ERROR] log lines detected in ${each.value}"
  comparison_operator = "GreaterThanOrEqualToThreshold"
  evaluation_periods  = 1
  metric_name         = "ApplicationErrors"
  namespace           = "ETLPipeline/Lambda"
  period              = 300
  statistic           = "Sum"
  threshold           = 5   # tolerate up to 4 errors per 5 minutes before alerting

  # No dimensions — the metric filter emits to the namespace without dimensions.
  # In production, add a "FunctionName" dimension to the metric_transformation
  # and filter here.

  alarm_actions      = [aws_sns_topic.ops_warning.arn]
  treat_missing_data = "notBreaching"

  depends_on = [aws_cloudwatch_log_metric_filter.app_errors]
}

Extended statistics (p99): The extended_statistic argument accepts percentile statistics like "p99", "p95", "p50". Use these for latency alarms instead of Average — averages hide tail latency that affects a small but significant percentage of users. Note: statistic and extended_statistic are mutually exclusive.

Step Functions Alarms

Step Functions emits metrics to the AWS/States namespace. The most important metrics for production monitoring are execution failures, timeouts, and throttles.

AWS/States
ExecutionsFailed
Executions that reached a Fail state or encountered an unhandled error. Any failure is critical for order/payment workflows.
AWS/States
ExecutionsTimedOut
Executions that exceeded the state machine's maximum duration. Indicates a stuck state or an infinite wait.
AWS/States
ExecutionThrottled
Executions rejected due to account-level or state machine concurrency limits.
AWS/States
ExecutionTime
Duration of completed executions in milliseconds. Alarm on p99 to detect slow workflows before they impact SLAs.

DynamoDB Alarms

DynamoDB emits metrics to AWS/DynamoDB. For PAY_PER_REQUEST tables, the most important signals are SystemErrors (DynamoDB-side failures) and UserErrors (client-side errors like invalid requests). For PROVISIONED tables, also alarm on ConsumedReadCapacityUnits and ConsumedWriteCapacityUnits approaching provisioned limits. 

# dynamodb_alarms.tf

# ── DynamoDB System Errors (per table) ────────────────────────────────
# SystemErrors indicate DynamoDB-side failures (HTTP 5xx). These are rare
# but critical — they indicate a DynamoDB service issue.
resource "aws_cloudwatch_metric_alarm" "dynamodb_system_errors" {
  for_each = toset(var.dynamodb_table_names)

  alarm_name          = "dynamodb-system-errors-${each.value}"
  alarm_description   = "DynamoDB table ${each.value} is returning system errors (5xx)"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 1
  metric_name         = "SystemErrors"
  namespace           = "AWS/DynamoDB"
  period              = 60
  statistic           = "Sum"
  threshold           = 0

  dimensions = {
    TableName = each.value
  }

  alarm_actions      = [aws_sns_topic.ops_critical.arn]
  treat_missing_data = "notBreaching"
}

# ── DynamoDB User Errors (per table) ──────────────────────────────────
# UserErrors indicate client-side errors (HTTP 4xx): invalid requests,

  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 2
  metric_name         = "UserErrors"
  namespace           = "AWS/DynamoDB"
  period              = 300

# ── DynamoDB Throttled Requests (per table) ────────────────────────────
# For PAY_PER_REQUEST tables, throttles indicate burst limits being hit.
# For PROVISIONED tables, they indicate under-provisioning.

Composite Alarms

Composite alarms combine multiple child alarms using boolean logic (AND, OR, NOT). They are the primary tool for reducing alert noise: instead of receiving 10 individual alarm notifications when a single root cause fires multiple alarms, you receive one composite alarm notification. Composite alarms do not evaluate metrics directly — they evaluate the states of other alarms.

Composite Alarm Design Pattern

Define fine-grained child alarms for each metric/function. Then define a composite alarm that fires when any critical child alarm is in ALARM state. Route only the composite alarm to PagerDuty — route individual child alarms to a lower-priority channel for context. 

# composite_alarms.tf

# ── Pipeline Health composite alarm ───────────────────────────────────
# Fires when ANY of the critical child alarms is in ALARM state.
# This is the single alarm to wire into PagerDuty or on-call rotation.
resource "aws_cloudwatch_composite_alarm" "pipeline_health" {
  alarm_name        = "pipeline-health-${var.environment}"
  alarm_description = "Overall pipeline health — fires when any critical alarm is active"

  # alarm_rule uses a boolean expression over child alarm names.
  # ALARM("alarm-name") evaluates to true when that alarm is in ALARM state.
  alarm_rule = join(" OR ", concat(
    # Lambda error alarms for all monitored functions.
    [for name in var.lambda_function_names :
      "ALARM(\"lambda-errors-${name}\")"],
    # Step Functions failure and timeout alarms.
    [
      "ALARM(\"sfn-execution-failures-${var.environment}\")",
      "ALARM(\"sfn-execution-timeouts-${var.environment}\")"
    ],
    # DynamoDB system error alarms for all monitored tables.
    [for name in var.dynamodb_table_names :
      "ALARM(\"dynamodb-system-errors-${name}\")"]
  ))

  # The composite alarm itself routes to the critical SNS topic.
  alarm_actions = [aws_sns_topic.ops_critical.arn]
  ok_actions    = [aws_sns_topic.ops_critical.arn]

  # Suppress individual child alarm actions when the composite is in ALARM.
  # This prevents duplicate notifications from both child and composite alarms.
  actions_suppressor                           = null
  actions_suppressor_wait_period               = null
  actions_suppressor_extension_period          = null
}

# ── Warning composite alarm ────────────────────────────────────────────
# Fires when any warning-level alarm is active.
resource "aws_cloudwatch_composite_alarm" "pipeline_warnings" {
  alarm_name        = "pipeline-warnings-${var.environment}"
  alarm_description = "Pipeline warnings — performance degradation detected"

  alarm_rule = join(" OR ", concat(
    [for name in var.lambda_function_names :
      "ALARM(\"lambda-throttles-${name}\")"],
    [for name in var.lambda_function_names :
      "ALARM(\"lambda-duration-${name}\")"],
    [
      "ALARM(\"sfn-execution-throttles-${var.environment}\")",
      "ALARM(\"sfn-execution-duration-p99-${var.environment}\")"
    ],
    [for name in var.dynamodb_table_names :
      "ALARM(\"dynamodb-throttles-${name}\")"]
  ))

  alarm_actions = [aws_sns_topic.ops_warning.arn]
  ok_actions    = [aws_sns_topic.ops_warning.arn]
}

Composite alarm billing: Composite alarms are billed per alarm per month, the same as standard alarms. However, they do not evaluate metrics themselves — they only evaluate child alarm states. This means they do not incur metric evaluation costs. Use them freely to reduce operational noise.

Operational Runbook Guidance

Alarms are only useful if responders know what to do when they fire. Document a runbook for each alarm. Below is a template for the most common alarm types in this module.

🔴 Lambda Errors Alarm

Immediate action: Check CloudWatch Logs for the function. Look for the exception type and stack trace. Common causes: IAM permission denied, DynamoDB throttle, SNS publish failure, malformed input event. Check the EventBridge dead-letter queue if configured.

🔴 Step Functions Execution Failed

Immediate action: Open the Step Functions console, find the failed execution, and inspect the execution history. The ExecutionFailed event contains the error and cause. Check the specific state that failed and its Lambda function logs.

🟡 Lambda Throttles Alarm

Immediate action: Check the account-level concurrency limit and the function's reserved concurrency setting. If the function is being throttled by EventBridge, the events will be retried automatically. Consider requesting a concurrency limit increase or adding reserved concurrency.

🟡 DynamoDB Throttled Requests

Immediate action: For PAY_PER_REQUEST tables, DynamoDB throttles burst traffic. Check if there is a hot partition key. For PROVISIONED tables, increase the provisioned capacity or enable auto-scaling. Review the access pattern and consider adding a GSI or changing the key schema.

Cleanup

terraform destroy
  • CloudWatch log groups are deleted by terraform destroy only if they are managed by Terraform. Log groups created automatically by Lambda (without a Terraform resource) must be deleted manually.
  • CloudWatch dashboards are deleted immediately — no data is retained.
  • Alarm history is retained in CloudWatch for 14 days after the alarm is deleted.
  • SNS subscriptions in PendingConfirmation state are automatically deleted after 3 days.

Common Pitfalls

Composite alarm rule references must be exact alarm names: The alarm_rule expression uses ALARM("alarm-name") where the name must exactly match the alarm_name of the child alarm. If a child alarm is renamed or not yet created, the composite alarm will fail to evaluate. Use depends_on to ensure child alarms exist before the composite alarm is created.

Log metric filter pattern syntax is not regex: CloudWatch filter patterns use a proprietary syntax. [ERROR] matches the literal string [ERROR]. ERROR (without brackets) matches any line containing the word ERROR. { $.level = "ERROR" } matches JSON logs. Do not use regex syntax — it will not work and may silently match nothing.

SNS topic policy must allow CloudWatch: CloudWatch alarm actions require the SNS topic policy to explicitly allow SNS:Publish from the cloudwatch.amazonaws.com service principal. Without this, alarm notifications are silently dropped — the alarm transitions to ALARM state but no message is sent.

  • Alarm evaluation delay: CloudWatch metrics have a 1–2 minute ingestion delay. An alarm with period = 60 and evaluation_periods = 1 may take 2–3 minutes to fire after the triggering event. For faster alerting, use shorter periods (minimum 10 seconds for high-resolution metrics).
  • treat_missing_data default: The default value is missing, which keeps the alarm in its current state when there is no data. For alarms where no data means no invocations (e.g., the "missed schedule" alarm), use breaching. For alarms where no data is normal (e.g., errors on a low-traffic function), use notBreaching.
  • extended_statistic vs statistic: These arguments are mutually exclusive. Using both in the same resource causes a Terraform validation error. Use statistic for standard statistics (Sum, Average, Maximum, Minimum, SampleCount) and extended_statistic for percentiles (p99, p95, etc.).
  • Dashboard widget metric arrays: The CloudWatch dashboard JSON uses a specific array format for metrics: ["namespace", "metric_name", "dimension_name", "dimension_value", {options}]. Incorrect array structure causes the widget to display "No data" without an error message.
  • Log group name must match Lambda's auto-created name: Lambda creates log groups named /aws/lambda/<function-name>. If your Terraform log group name does not match exactly, you will have two log groups — one managed by Terraform (with retention) and one created by Lambda (with infinite retention).
  • for_each with alarm names in composite rules: When using for_each to create per-function alarms, the composite alarm's alarm_rule must reference the exact alarm names generated by for_each. Use Terraform's join() and list comprehensions (as shown) to build the rule dynamically.

Best practice — alarm naming convention: Use a consistent naming pattern: <service>-<metric>-<resource-name>-<environment>. This makes alarms sortable and filterable in the CloudWatch console, and makes composite alarm rules readable. Avoid spaces and special characters in alarm names.

Best practice — alarm-to-runbook linking: Set the alarm_description to include a link to the runbook for that alarm (e.g., a Confluence page or GitHub wiki URL). When an on-call engineer receives an SNS notification at 3 AM, the runbook link in the alarm description is the fastest path to resolution.
